### 1. The Database Specification (Supabase PostgreSQL)

Table Name: **`MarketplaceItems`**

| Column Name | Type | Required? | Description |
| :--- | :--- | :--- | :--- |
| `id` | `UUID` | Yes | Generated by Supabase (Primary Key). |
| `created_at` | `TIMESTAMPTZ` | Yes | Timestamp of creation. |
| `user_id` | `UUID` | No (for now) | Link to Auth later (anonymous/hardcoded for the MVP). |
| `title` | `TEXT` | Yes | Name of the object. |
| `file_path` | `TEXT` | Yes | Path to the video in the `raw_scans` bucket (e.g., `id/video.mp4`). |
| `kiri_serialize` | `TEXT` | No | The unique ID returned by KIRI after upload. |
| `used_api_key_id` | `UUID` | Yes | Foreign key to `apiproviderkeys.id`, tracking which KIRI account handled this item. |
| `model_url` | `TEXT` | No | The public download link of the final `.zip`. |
| `status` | `item_status` | Yes | Enum: `UPLOADING`, `SENDING_TO_KIRI`, `PROCESSING_IN_CLOUD`, `DOWNLOADING_ARTIFACT`, `READY`, `FAILED`. |

Table Name: **`apiproviderkeys`** (lowercase)

| Column Name | Type | Required? | Description |
| :--- | :--- | :--- | :--- |
| `id` | `UUID` | Yes | Primary key for each provider key record. |
| `api_key` | `TEXT` | Yes | Raw KIRI API key string used by Edge Functions. |
| `usage_count` | `INT` | Yes | Number of scans consumed on this key. |
| `max_limit` | `INT` | Yes | Maximum scans for this key (default `20`). |
| `is_active` | `BOOL` | Yes | Whether this key can be selected for new jobs. |

Security for `apiproviderkeys`:
* RLS is **enabled**.
* Access for `anon` and `authenticated` is fully revoked.
* Only Edge Functions (service role context) are allowed to read/write this table.

Database RPC Function: **`get_available_credits()`**
* Type: `SECURITY DEFINER`
* Returns: a single integer
* Logic: `sum(greatest(max_limit - usage_count, 0))` across active keys
* Purpose: exposes safe credit totals to the app without exposing actual API keys.

### Why the Multi-API-Key System Exists

KIRI gives about 20 free generations per account (1 credit = 1 model), and then immediately hits us with a minimum top-up of 500 credits (~500 USD).
For an MVP, that is less "small experiment" and more "who approved this enterprise invoice?"
So yes, the API key pool is both a technical design choice and a survival strategy for our student-level wallet.
To add API keys to the pool, we create a bunch of Kiri accounts and add their API key to this table.
If we ever run out of tokens, we just create a new account and add the api key to the table, no code change needed, peanuts.

The `apiproviderkeys` table introduces a rotation pool so the backend can:
* spread scan usage across multiple KIRI accounts,
* keep the MVP running longer without manual key swaps,
* isolate failures (one inactive/exhausted key does not block all traffic),
* keep API secrets out of the Android client.

### How Key Rotation Works in Practice

1. `start-kiri-job` selects one row from `apiproviderkeys` with `is_active = true` and `usage_count < max_limit`.
2. That selected key is used to submit the upload to KIRI.
3. On success, the function:
   * stores `used_api_key_id` in `MarketplaceItems`,
   * increments that key's `usage_count`.
4. Later, `kiri-webhook` uses `used_api_key_id` to fetch the exact same API key again for `getModelZip`.

This attribution (`used_api_key_id`) is critical: without it, webhook downloads could fail or be inconsistent if different KIRI accounts are mixed.

### Security Boundary (Important)

* Android app access is limited to anonymous REST/RPC calls.
* Android never reads `apiproviderkeys` directly.
* `apiproviderkeys` is service-role only (Edge Functions), protected by RLS + revoked anon/authenticated access.
* Android sees only aggregate credits through `get_available_credits()`.

### 2. Storage Specification (Supabase Storage)

* **Bucket 1: `raw_scans`**
    * **Visibility:** Private (Only accessible to logged-in app/Edge Functions).
    * **Content:** The original `.mp4` videos.
* **Bucket 2: `3d_models`**
    * **Visibility:** Public (or at least readable for the app).
    * **Content:** The final `.zip` files (which contain the `.ply` and `cameras.json`).

### 3. The Pipeline: Step by Step

#### Step A: Recording & Upload (Android -> Supabase)
1.  **Recording (Android CameraX):** * *Requirements:* The app enforces that the video is a maximum of 1920x1080 (1080p) and lasts a maximum of 3 minutes (KIRI hard limits).
2.  **Database Registration:** The Android app creates a row in `MarketplaceItems`.
    * *Action:* Insert with `title` = "Object X", `status` = `UPLOADING`.
    * *Result:* App receives the generated `id` (UUID).
3.  **Storage Upload:** The app uploads the video to the `raw_scans` bucket.
    * *Path:* `[id]/video.mp4`.
4.  **Completion:** The app updates the database row to populate `file_path` with `[id]/video.mp4`.

#### Step B: Trigger the Upload to KIRI (Android -> Edge Function 1)
1.  **Invocation:** The app sends a POST request to the `start-kiri-job` Edge Function.
    * *Body:* `{"item_id": "[id]"}`
2.  **Processing in the Edge Function:**
    * Locks the database (sets status to `SENDING_TO_KIRI`).
    * Downloads the video from `raw_scans` into internal memory.
    * Selects an API key from `apiproviderkeys` where `is_active = true` and `usage_count < max_limit`.
    * Builds a `FormData` object with:
        * `isMesh` = `"0"`
        * `isMask` = `"1"` (enables KIRI background masking)
        * `videoFile` = the blob of the `.mp4`.
    * Sends a POST request to `https://api.kiriengine.app/api/v1/open/3dgs/video` with the header `Authorization: Bearer [selected_api_key]`.
3.  **Saving:** On success, updates `MarketplaceItems` with `kiri_serialize` = `[serialize_id]`, `used_api_key_id` = `[selected_key_id]`, and `status` = `PROCESSING_IN_CLOUD`, then increments `usage_count` for that selected key.
4.  **If No Key Is Available:** If no row matches (`is_active = true` and remaining quota), the function returns a credits-exhausted response and the app disables generation until capacity is restored.

#### Step C: Rendering & Waiting (HTTP Polling)
* **Android Side:** The app uses raw HTTP (`OkHttp`) polling to check status. It sends `GET /rest/v1/MarketplaceItems?id=eq.[id]&select=status` roughly every 10 seconds until status becomes `READY` or `FAILED`.
* **KIRI Side:** KIRI Engine calculates the Gaussian Splatting in their cloud (takes approx. 10-15 mins).

#### Step D: Webhook Reception (KIRI -> Edge Function 2)
1.  **The Trigger:** KIRI is finished and fires a POST request to your `kiri-webhook` Edge Function.
    * *Header:* KIRI sends `X-Signature: [your_password]`.
    * *Body:* `{"status": 2, "serialize": "[serialize_id]"}`.
2.  **Security:** The Edge Function checks if `X-Signature` exactly matches the `KIRI_WEBHOOK_SECRET`. If not: block it (403).
    * **Important:** Every KIRI account in the rotation pool must use the exact same webhook secret in its KIRI dashboard.
3.  **Searching & Locking:** The function searches `MarketplaceItems` by `kiri_serialize`, retrieves the row, and sets status to `DOWNLOADING_ARTIFACT`.
4.  **Resolve Key Dynamically:** Using `used_api_key_id`, the function reads `apiproviderkeys` to load the exact API key string used for this scan.

#### Step E: Downloading & Saving (Edge Function 2 -> Supabase Storage)
1.  **Fetch Link:** The function makes a GET request to `https://api.kiriengine.app/api/v1/open/model/getModelZip?serialize=[serialize_id]` with the dynamically resolved API key from Step D.
2.  **Pump the Stream:** The function downloads the `.zip` from KIRI's `modelUrl` and uploads it directly as a stream into your `3d_models` bucket at the path `[id]/model.zip`.
3.  **Completion:** The function fetches the public URL of the zip, updates the database: `status` = `READY`, and `model_url` = `[public URL]`.

#### Step F: Client Signal & Display (Supabase -> Android)
1.  **UI Credit Check (Main Menu):** On menu load, the app calls `POST /rest/v1/rpc/get_available_credits` and displays `Available Scans: [count]`. If count is `<= 0`, the camera/record button is disabled.
2.  **Storage Safety Check (Before Download):** Before downloading a model zip, the app checks `context.cacheDir.usableSpace`. If below `250 * 1024 * 1024` bytes, the download is aborted and the user is warned.
3.  **Smart Cache Workflow (Marketplace UI):**
    * Models are stored only in `cacheDir/models/[item_id]/model.ply`.
    * The app checks `File(cacheDir, path).exists()` to decide whether each item shows a `Download` button or a `View 3D Model` button.
    * On download: save zip to cache, extract `.ply` to cache, then delete the zip immediately.
    * This keeps storage lean and relies on Android OS cache eviction for old files (self-healing offline behavior).
4.  **The Viewer (Android WebView):**
    * The app opens a hidden HTML page in a WebView where `gsplat.js` (or WebXR viewer) is integrated.
    * The app passes the local file path of the extracted `model.ply` to the WebView.
    * The 3D Gaussian Splat appears smoothly and manipulable on the user's screen.
